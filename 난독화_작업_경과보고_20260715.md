# 네이버 블로그 자동화 — 소스 난독화 작업 경과 보고 (2026-07-15)

## 1. 목적

배포되는 앱(Electron)의 소스코드(main.js, preload.js, license 검증 로직, DB 로직)를 AI 도구 등에 넣어 복사본을 만들거나 무료로 수정해 쓰는 것을 막기 위해, 배포 빌드에서만 소스를 난독화하는 파이프라인을 구축.

- 원본 소스는 절대 수정하지 않음 — 개발자가 평소 여는 파일은 그대로 유지.
- 배포 빌드(패키징) 직전에만 자동으로 난독화된 사본을 만들어 그걸 패키징에 사용.
- 우선순위: **보안 강도보다 설치·실행 안정성이 우선**(사용자 명시적 확인).

## 2. 구현 개요

- `scripts/obfuscate.js`: `javascript-obfuscator` 패키지로 main.js/preload.js/src/db.js/license/*.js를 읽어 `build-obfuscated/`(gitignore 대상)에 난독화된 사본 생성.
- `scripts/beforePack.js`: electron-builder의 `beforePack` 훅으로 등록 — 패키징 직전 자동 호출.
- `package.json`의 `build.files`: `{from:"build-obfuscated", to:".", filter:["**/*"]}`로 원본 대신 난독화 사본을 패키징.
- 이 설정은 **플랫폼 무관**(top-level) — Mac/Windows 빌드 모두에 동일하게 적용됨.

## 3. 발견·수정된 버그 4건 (모두 2026-07-15, 순차 발견)

| # | 증상 | 원인 | 조치 |
|---|------|------|------|
| 1 | 패키징 마지막 단계에서 "Application entry file main.js ... does not exist" | electron-builder 24.13.3에서 개별 파일 rename FileSet(`{from:"build-obfuscated/main.js", to:"main.js"}`)이 실제로 복사 안 됨 | 디렉터리 전체를 통째로 매핑하는 방식(`{from:"build-obfuscated", to:".", filter:["**/*"]}`)으로 교체 |
| 2 | Mac universal(x64+arm64) 병합 시 "Can't reconcile two non-macho files main.js" | beforePack이 아키텍처마다 별도 실행되는데, javascript-obfuscator가 매번 다른 결과를 냄(무작위 식별자 배치) | 고정 seed + mtime 기반 캐시(같은 빌드 내 재사용) 추가 |
| 3 | 앱 실행 즉시(3초 내) macOS 크래시(EXC_BREAKPOINT, V8 내부) | selfDefending/controlFlowFlattening/deadCodeInjection이 원인으로 의심됨(Electron 호환성 문제 + 300KB대 파일에서 V8 컴파일러 부담) | 이 3개 옵션 비활성화, stringArray/식별자치환은 유지 |
| 4 | 버그#3 수정 후 재빌드해도 캐시가 "재사용"만 표시 — 수정사항이 실제 반영 안 됨 | 캐시가 파일 수정 시각(mtime) 비교 방식이었는데, 이 시각 정보가 정확히 동기화 안 됨 | 캐시 판단을 **콘텐츠 SHA-256 해시** 비교로 전면 교체(`.obfuscate-cache/`) — 시각 정보에 전혀 의존하지 않음 |

버그 4 수정 후 재빌드 로그에서 실제로 "새로 생성"(해시 불일치로 재생성)이 확인됐고, 이후 "해시 일치 — 재사용"이 뜨는 빌드는 안전 옵션이 실제로 적용된 결과물임이 보장됨.

## 4. 실행 크래시 원인 조사 — 총 5회 테스트

버그#3 수정(옵션 3개 비활성화)이 실제로 반영된 빌드를 설치해도 **크래시가 동일하게 재현**되어, 추가로 원인을 격리하는 테스트를 5차례 진행. 모든 크래시는 macOS 크래시 리포트 기준 **정확히 같은 지점**(`v8::BackingStore::MaxByteLength() const + 12292`, `imageOffset 3599184`, `CrBrowserMain` 스레드, `EXC_BREAKPOINT`/`brk 0`, 앱 실행 후 약 3초)에서 발생.

| 테스트 | 조건 | 결과 |
|--------|------|------|
| 1 | selfDefending 등 3개 옵션 ON (최초) | 크래시 |
| 2 | 3개 옵션 OFF (mtime 캐시로 실제 미반영 의심) | 크래시(동일 지점) |
| 3 | 3개 옵션 OFF, 해시 캐시로 반영 확인된 빌드 | 크래시(동일 지점) |
| 4 | `SKIP_OBFUSCATION=1` — 원본 코드 그대로, 난독화 완전 미적용 | 크래시(동일 지점) |
| 5 | mac universal → arm64 전용 빌드(병합 과정 배제) | 크래시(동일 지점) |

**결론: 크래시는 난독화(옵션 포함 전체)와도, mac universal 병합 과정과도 완전히 무관함.** 원인은 이 앱의 코드나 빌드 설정이 아님.

## 5. 외부 리서치 결과

웹 검색으로 확인한 결과, 이 크래시는 **Electron 생태계 전반의 알려진, 아직 미해결인 이슈**로 확인됨.

- GitHub `electron/electron` 이슈 [#49522](https://github.com/electron/electron/issues/49522)("EXC_BREAKPOINT crash on macOS 26.2 Tahoe during ElectronMain initialization")와 [#51351](https://github.com/electron/electron/issues/51351)에서 동일 패턴(`v8::BackingStore::MaxByteLength`류 심볼, ElectronMain 초기화 중 앱 JS 실행 전 크래시)이 다수 보고됨.
- 재현 범위: Electron 28.3.3 ~ 41.x 사이 여러 버전에서 공통 재현 — 즉 Electron 버전을 바꿔도 해결 안 될 가능성 높음. macOS 26(Tahoe) 자체와의 상호작용 문제로 추정.
- 중요 단서(이슈 #49522 댓글): 같은 코드·같은 Electron 버전이라도 **"MAS(앱스토어)+App Sandbox → 크래시 / Developer ID 서명(비샌드박스) → 정상 / `npx electron .`(dev 모드) → 정상"**으로 나뉜다는 실측 보고가 있음. 즉 "패키징된 .app의 서명 방식"이 크래시 여부를 가르는 핵심 변수일 가능성.
- 이 이슈들은 검색 시점 기준 **미해결 상태**이며 Electron 팀도 명확한 픽스를 내놓지 못한 것으로 보임.

### 실사용 환경 정보
- Electron 33.4.11 / electron-builder 24.13.3
- macOS 26.5.2 (빌드 25F84, "Tahoe" 계열)
- Apple M4 Pro, Mac16,8, 48GB RAM
- 이 앱은 현재 **코드사이닝을 전혀 하지 않는 상태**(0 valid identities found) — 위 "3가지 경우" 중 정확히 어디에 해당하는지는 실측으로만 확인 가능.

## 6. 확정 실험 — dev 모드는 정상 작동

`npm start`(패키징 없이 `electron .`로 바로 실행)는 **크래시 없이 정상 작동함**을 사용자가 직접 확인. 이는 "패키징(서명 방식)이 변수"라는 커뮤니티 보고 패턴과 정확히 일치. 동시에, 사용자가 평소 dev 모드로만 작업해왔고 실제 패키징+설치까지 끝까지 테스트한 게 이번 난독화 작업이 계기로 처음이었을 가능성이 크다는 것도 확인됨 — 즉 "이번에 새로 생긴 회귀"가 아니라 "macOS가 26으로 업데이트된 이후 원래부터 있었지만 지금까지 발견되지 않았던 문제"로 추정.

## 7. 사용자 결정

두 가지 선택지를 제시:
1. 실제 Apple Developer ID 인증서로 정식 서명+공증(연 $99, Apple Developer Program 가입) — 커뮤니티 보고상 효과가 있을 수 있으나 100% 보장은 아님.
2. 다른 Mac/구버전 macOS에서 패키징된 앱이 정상 동작하는지 먼저 확인.
3. Electron/Apple 업스트림 수정을 기다리며 **macOS 대상 배포는 당분간 보류**.

→ **사용자는 3번(당분간 배포 보류)을 선택.** macOS 배포는 보류하되, 난독화 파이프라인 자체는 구현이 정상 완료된 상태이며, 크래시는 별개 문제로 분리해 기록.

## 8. 현재 진행 상황 — Windows 빌드로 전환

Mac 사용자보다 Windows 사용자가 실제로 더 많다는 판단 하에, Windows 빌드에 집중하기로 함. 난독화 파이프라인 자체는 플랫폼 무관(공통 설정)이라 Windows용으로 별도 작업이 필요 없음 — 단, 이번 세션엔 Windows 빌드를 한 번도 실제로 테스트하지 않아 별도 검증 필요.

### 부수적으로 발견·수정한 문제
- git push 시 `dist/`(electron-builder 빌드 산출물, 수백MB) 폴더가 `.gitignore`에서 빠져 있어 커밋에 통째로 딸려 들어감 → GitHub 100MB 파일 크기 제한으로 push 거부. `.gitignore`에 `dist/` 추가 + `git commit --amend`로 정리하여 해결·push 완료.

### 현재 막힌 지점
Windows PC(별도 보유)에 프로젝트를 처음 설치하는 중 — 이 PC에 **Git과 Node.js가 아예 설치되어 있지 않음**을 확인. 사용자에게 Git for Windows, Node.js(LTS) 설치 안내 중. 1차 설치 후 `npm`은 인식됐으나 `git`은 아직 인식 안 되는 상태(Git 설치 미완료 또는 PATH 미반영 추정) — 재설치/PowerShell 재시작 안내 후 결과 대기 중.

## 9. 남은 과제 요약

1. Windows PC에 Git/Node.js 정상 설치 확인 → 프로젝트 clone → `npm install` → `npm run build:win`으로 실제 빌드 및 설치 테스트.
2. Windows 빌드가 정상 동작하면, Windows 사용자 대상으로 이번 난독화 릴리즈를 먼저 배포하는 방향 검토.
3. Mac 크래시는 별도 트랙으로 보류 상태 유지 — Electron/Apple 쪽 업스트림 수정 여부를 주기적으로 확인할 필요 있음(다음에 다시 논의 시).
